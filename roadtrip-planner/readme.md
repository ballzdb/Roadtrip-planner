Description:
Road Trip Planner is a command-line Python program that helps someone plan a road trip. You type in two or more cities, pick what kind of car you're driving, and the program tells you how far it is, how long it will take, and roughly how much you'll spend on gas. If you type in more than two cities, it also figures out the smartest order to visit them in, instead of just going in the order you typed them.

I built this as my CS50 final project. I didn't want it to just be a simple script that calls an API and prints a number. I wanted it to actually do some thinking, so I added a couple of real algorithms into it, which I'll explain below.

What the program actually does
First, you type in your cities one at a time. The program uses a geocoding API (OpenRouteService) to turn each city name into real coordinates, since you can't calculate distance or get directions from just a name.

If you only typed two cities, the program grabs the real driving route between them and tells you the distance and how long it'll take. It also does something extra here: it calculates the straight-line distance between the two cities using something called the Haversine formula. This is just math that figures out the distance between two points on a sphere, since the Earth isn't flat and you can't just use regular geometry. Once it has both numbers, it shows you the difference as a percentage, which I call "road overhead." For example, LA to San Diego only has about 10% overhead, because the road is basically straight along the coast. But Denver to Salt Lake City has over 40% overhead, because the road has to twist through mountains instead of going in a straight line. I liked this feature because it turns a boring distance number into something that actually tells you a bit about the geography between two places.

If you type in more than two cities, the program has to figure out the best order to visit them, which is a classic computer science problem called the Travelling Salesman Problem. Basically, if you have a bunch of cities, there are a huge number of possible orders you could visit them in, and you want the one that adds up to the shortest total distance. My program solves this in two different ways. If there are 8 or fewer cities, it tries literally every single possible order and picks the shortest one. This is called brute force, and it always finds the actual best answer, but it gets slow extremely fast, since the number of possible orders grows factorially. With 8 cities that's already over 40,000 combinations, and by 12 cities it's in the hundreds of millions. So for anything bigger than 8 cities, the program switches to a different method called nearest neighbor, where it just keeps going to whichever city is closest that it hasn't visited yet. This isn't always the perfect answer, but it's way faster and can handle any number of cities without freezing your computer.

Once the program knows the best order to visit the cities, it calls the routing API again for each real segment of the trip, adds up the total distance and time, and grabs the current average gas price in the US from a government website called fueleconomy.gov. Then it does simple math with your car's MPG to estimate how much the whole trip will cost you in gas.
